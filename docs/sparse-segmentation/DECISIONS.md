# Decisions

This file records locked decisions for the sparse segmentation hard cutover. Do not reopen these decisions casually during implementation.

## D1: Segmentation is a sparse categorical field

Segmentation layers are no longer modeled as dense volumes. They are sparse categorical voxel fields made of non-empty bricks.

Consequences:

- No segmentation layer should expose a dense global `labels` array.
- No segmentation renderer should require a full-volume label texture.
- Provider APIs must distinguish dense intensity volumes from sparse segmentation fields.
- Shared viewer code may keep an `isSegmentation` convenience flag, but core data loading must not treat segmentation as a dense volume variant.

## D2: Hard cutover for segmentation

Legacy dense segmentation datasets are unsupported.

Allowed:

- old or current preprocessed datasets with no segmentation layers
- newly preprocessed datasets with sparse segmentation layers

Rejected:

- any preprocessed manifest that contains a segmentation layer using the old dense zarr data representation
- any segmentation layer that lacks the new sparse segmentation representation
- any runtime path that silently loads dense segmentation as a fallback

The error must occur before launching the viewer.

## D3: Intensity-only compatibility remains

The hard cutover applies to segmentation only.

Older preprocessed datasets that contain only intensity channels must still run. This project is not using this refactor to break intensity-only data.

Schema and import code must therefore reject by layer representation, not only by root manifest format.

## D4: No geometry conversion

Segmentation must not be converted to meshes, marching-cubes surfaces, point clouds, splats, instanced cubes, or other geometry representations.

All segmentation rendering remains voxel-based.

## D5: WebGL2 first, WebGPU later

The implementation must work on the existing WebGL2 + Three.js stack.

The design should still keep these boundaries clean:

- storage format
- CPU provider/query APIs
- renderer resource preparation
- shader backend

Future WebGPU work should be able to replace the WebGL2 resource/shader backend without changing the on-disk sparse segmentation format.

## D6: No dense global expansion

Preprocessing, loading, rendering, slice extraction, hover, and tests must not expand a segmentation layer into a full dense global label volume as the normal path.

Small dense reference arrays are allowed only inside tests for expected-output comparison.

## D7: Sparse bricks are the canonical storage unit

The canonical sparse representation is a multiscale brick hierarchy:

- fixed-size brick grid per timepoint and scale
- empty bricks omitted
- non-empty bricks encoded into payload shards
- compact brick directory/index metadata
- label metadata per layer and per scale

This is preferred over one global list of nonzero voxels because rendering, slice extraction, hover lookup, and cache residency all need spatial locality.

## D8: Default base brick size is 32 x 32 x 32

The default preprocessing brick size is 32 voxels per axis.

Rationale:

- local linear voxel offsets fit in `uint16`
- GPU local brick atlas payloads stay manageable
- brick-level culling is fine-grained enough for sparse biological segmentations
- WebGL2 texture and loop constraints are easier than with 64-cubed bricks

The manifest may allow other power-of-two brick sizes after validation, but the first complete implementation should optimize for 32-cubed bricks.

## D9: Logical label type is uint32

The sparse segmentation system uses `uint32` logical label IDs.

Rules:

- label `0` is background
- nonzero labels are foreground objects or regions
- labels must be finite non-negative integers
- floating non-integer label values fail preprocessing
- labels above `uint32` range fail preprocessing

This removes the current `uint16` ceiling from segmentation.

## D10: Downsampling is categorical

Segmentation downsampling must never average labels.

For each output voxel, choose the source label by majority vote over the covered source voxels. Ties are resolved deterministically:

1. highest voxel count wins
2. if tied, nonzero wins over zero
3. if still tied, the smaller numeric label wins

This rule must be pinned by tests.

## D11: Missing resident bricks are not empty

If a brick is occupied according to the sparse directory but its payload is not resident in CPU or GPU memory, it must not be treated as empty.

The viewer must either:

- block the layer's render-ready state until required visible bricks are resident, or
- render with an explicit incomplete/loading state that cannot be mistaken for correct empty space

Silent false negatives are correctness bugs.

## D12: Label colors are hash-based

Segmentation colors are generated by deterministic hash from:

- layer color seed
- label ID

The renderer must not depend on a fixed 65,536-entry palette. A small palette texture may still be used as an implementation cache only if it supports `uint32` labels through hashing or indirection.
